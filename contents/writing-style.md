# 技術記事執筆ガイドライン

## 本ガイドラインの目的と基本方針

### 目的
本ガイドラインは、読者にとって理解しやすく、信頼性の高い技術記事を執筆するための共通指針を定めることを目的とします。

### 基本方針：論調の使い分け
技術記事は、対象読者や内容に応じて2つの論調を使い分けます。

- **硬めの論調:** 初めてその技術に触れる読者でも理解できるよう、前提知識から丁寧に解説します。背景や歴史的経緯を含め、知識の土台を固めることを重視します。
- **ふわっとした論調:** 特定のアップデート情報や、簡単な「試してみた」系の記事など、前提知識の解説が冗長になる場合は、要点を絞った簡潔な構成を目指します。

---

## 執筆の核となる思想：コンテキストファースト

我々が執筆する記事は、単なる情報の羅列であってはなりません。読者の理解を最大化するため、**コンテキストファースト**の原則を最も重要視します。

### なぜコンテキストが重要か
人間は、大量の情報を処理する際に**ストーリー形成を通じて理解・認知します**。記事の導入部で読者と執筆者の間の前提知識（コンテキスト）を揃えることで、記事全体の理解度が飛躍的に高まります。

この考え方は、Stack Overflowが常に問題の背景を説明してから解決策を提示する構成になっていることからも、その有効性がわかります。

### コンテキストを揃えるための導入部
導入部の役割は、**Situation（状況） → Complication（複雑化・課題） → Question（疑問・本題）** というストーリーを提示し、読者の関心を引きつけ、前提知識を揃えることにあります。
- **コンテキストの具体例:**
  - 「このエラーは、〇〇というライブラリをバージョンアップした際に発生しました」（状況）
  - 「この技術は、従来の〇〇が抱えていたパフォーマンス問題を解決するために生まれました」（背景）

### 歴史解説からの学び
YouTubeの「大人の教養TVchannel」が過去日本が占領した地域の動画を解説する際に、必ず「昔日本の領土だった」という歴史的背景を説明するように、技術記事においても扱う技術が生まれた背景や前提となる知識を最初に提示することが、読者の深い理解に繋がります。

## 記事の基本構成テンプレート

記事の構成は、上記のコンテキストファーストの思想に基づき、以下のテンプレートを基本とします。

1.  **導入部:** コンテキストを揃える (Situation → Complication → Question)
2.  **課題提起:** 記事が解決しようとしている具体的な問題
3.  **解決策:** 課題を解決するための技術、ツール、手法の紹介
4.  **実装例:** 解決策を具体的に示すコードや手順
5.  **結論:** 記事全体の要点をまとめ、学びやクイズを提示
6.  **参考文献:** 記事の信頼性を担保するための一次情報や公式ドキュメント

## 記事パターンの分類とコンテキスト適用の指針

執筆する記事を以下の3パターンに分類し、「硬めの論調」か「ふわっとした論調」かを判断します。

1.  **新仕様の解説記事（硬めの論調）**
    - **内容:** 既存の技術に追加された新しい仕様（例: Claude CodeのSkillなど）。
    - **コンテキスト:** **必須。** 「既存技術とは何か」「なぜこの仕様が追加されたのか」といった背景説明が不可欠です。

2.  **既存技術の活用記事（硬めの論調）**
    - **内容:** 複数の技術を組み合わせた応用的な内容。
    - **具体例:** 
        - [GASを使うと社内だけで使えるWebサイトが簡単に作れるの知ってました？ - Zenn](https://zenn.dev/sui_water/articles/502a6ef30b79e1)
        - [Anthropic公式のMCPによるコード実行をserena MCPで試してみたけど無理だった件 - Zenn](https://zenn.dev/sui_water/articles/717eaa7e5b2425)
    - **コンテキスト:** **必須。** 「なぜこの技術を選んだのか」「どのような課題を解決するのか」といった土台の共有がなければ、読者は記事についていけません。

3.  **Tips・調査記事（ふわっとした論調）**
    - **内容:** 少し気を楽にした解説や、特定のエラー解決策、小規模な実践報告。
    - **具体例:**
        - [Notionの破壊的変更の記事](https://zenn.dev/sui_water/articles/717eaa7e5b2425)
        - [PreToolUseがfailed with non-blocking status code 127になったときにやったこと - Zenn](https://zenn.dev/sui_water/articles/ce32e569d738ea)
    - **コンテキスト:** **任意。** 記事の目的が特定の情報提供に絞られている場合、「NotionAPIとは？」のような冗長な説明はかえって読者の負担になります。

## 詳細スタイルガイド (Writing Style)

### 全体的な特徴
- **文体:** 基本的に日本語の「ですます調」を用いた丁寧語で記述します。
- **トーン:** 客観的かつ丁寧な解説を主としますが、自身の経験や考察を述べる際には「〜と思います」「〜と感じました」といった主観的なトーンを含んでも構いません。
- **対象読者:** Web開発技術に関心のあるエンジニアを想定しつつも、記事ごとに想定読者レベルを意識します。

### フォーマット・表現技法
- **コードブロック:**
    - 言語指定 (`js`, `ts`, `sh`, `diff` など) を必ず行います。
    - コードの差分を示す場合は `diff` 構文を効果的に使用します。
- **視覚要素:**
    - スクリーンショットや図解などを効果的に使用し、視覚的な理解を助けます。
- **リスト・表:**
    - 箇条書き (`-`, `*`) や番号付きリスト (`1.`)、Markdownテーブルを用いて情報を整理します。
- **強調:**
    - 太字 (`**text**` or `<b>text</b>`) やインラインコード (`` `code` ``) を使用して、重要なキーワード、ファイル名、コマンドを強調します。
- **リンク:**
    - 公式ドキュメントや関連技術、参考記事へのリンク (`[text](url)`) を積極的に活用します。
- **注意喚起:**
    - `>[CAUTION]` や `>[NOTE]`などのコールアウトを用いて、注意点、補足情報などを目立たせます。

### サンプルコードの品質と再現性
読者の学習体験を最大化するため、掲載するコードは以下の基準を満たすよう努めます。
1.  **再現性への配慮:**
    - 読者が手元でコードを再現できるよう、関連するライブラリのバージョンや実行環境、前提となる設定などを明記します。
    - 可能であれば、完全なプロジェクトをGitHubリポジトリで公開し、リンクを記載することを推奨します。
2.  **可読性の追求:**
    - 変数名や関数名は、その役割が理解しやすいように命名します。
    - 記事の本質と関係のない定型的なコードは適宜省略し、読者が最も重要なロジックに集中できるよう配慮します。
3.  **セキュリティへの言及:**
    - コード例がセキュリティ上の配慮（APIキーの管理など）を必要とする場合、そのリスクと対策について注意喚起を行います。

### 語彙・文法と推奨ルール
- **一人称:** 「私（わたし）」を使用します。
- **接続詞:** 「しかし」「そのため」「一方で」などを効果的に使用し、論理的な流れを明確にします。
- **一次ソースの重視:**
    - 一般的な常識を除き、技術的な記述には必ず一次ソース（公式ドキュメント、GitHubの実装情報など）をあたり、併記します。
    - 客観的な事実と、個人の意見や考察を明確に区別して記述します。
- **MECEと文末表現:**
    - 箇条書きの項目は、網羅的で重複がない状態（MECE）を意識します。
    - **通常文:** 同じ語尾（「〜です」など）が3回以上連続しないよう、文末表現に変化を持たせ、文章にリズムを生みます。
    - **箇条書き:** 逆に語尾を統一することで、情報の整理を容易にします。
    - 同種の項目が3つ以上ある場合は、文章で列挙するのではなく、箇条書きを使用します。
        - 悪い例：今から話すのはカレーに欠かせない材料です。一つ目は新米で炊き立て白ごはんです。二つ目は質の高い牛肉です。三つ目は隠し味のりんごです。
        - 良い例：■ カレーで重要な材料 ・新米で炊き立て白ごはん ・質の高い牛肉 ・隠し味のりんご

### その他
- **フロントマター:** YAML形式で記事のメタデータを詳細に管理します。
- **Textlint:** `.textlintrc.json` の存在から、日本語技術文書向けの校正ルールを適用し、文章の品質維持に努めていることがうかがえます。
